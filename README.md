UAM-RL-Scheduler-IncomingAware

UAM 垂直起降场调度仿真研究
城市平均地面速度 20 km/h | 单乘客 eVTOL 容量 | 8 起降场 | X 型布局
排队预测强化学习 (PPO) | 最优 ATT: 35.68 分钟


一、项目摘要

UAGMC 论文（IEEE TITS 2026）首次提出了端到端的空地协同空中出租车调度框架，
通过 MSCE 多源嵌入与 STIN 时空 Transformer 网络处理高维异构状态信息，验证
了深度强化学习在 UAM 起降场选择中的有效性。然而，真实城市交通环境呈现车速
低（市区平均 20-30 km/h）、起降场数量多、乘客出行距离分布及需求差异大等
特征，现有方法在贴近实际场景下的性能表现与计算开销仍需进一步探索。

基于上述问题，本项目在 UAGMC 框架基础上构建了一个符合未来城市交通条件的
UAM 起降场调度仿真平台，核心思路为：

  (1) X 型 8 起降场网络，覆盖 40x40 km 城市区域，增强对多方向出行需求的
      空间感知能力
  (2) 出行距离感知的乘客分类机制，乘客在地图内随机刷新，对短途出行乘客
      直接分配地面交通，避免强行分配飞机导致的效率损失
  (3) 真实 eVTOL 调度动力学仿真，制定空闲 eVTOL 的调度策略，将服务率近似
      替换为多机协同的全生命周期调度
  (4) 排队预测强化学习，在状态空间中引入"飞往起降场的飞机数量"特征，
      使智能体具备排队趋势预判能力，并采用七选一动作空间为飞行乘客选择
      合适的起降场

实验结果表明，在 20 km/h 城市平均速度、单乘客 eVTOL 容量、8 起降场的场景
下，轻量级 PPO+MLP 模型有效降低了系统平均出行时间与高峰时段排队压力，
验证了精简特征和动作空间裁剪在更真实的城市场景中的有效性，实现了 UAM 系统
的轻量化调度策略。


二、问题建模

2.1 出行链建模

本文研究一个集成了地面 CAV 与空中 eVTOL 的多模式乘客运输系统。每位乘客的
出行包含四个阶段：

  起点 -> CAV -> 起飞机场 -> 排队 -> eVTOL -> 飞行 -> 降落机场 -> CAV -> 终点

每位乘客 i 属于 N 具有起点 o_i 和终点 d_i，需要选择起飞机场 k_i 属于 V
和降落机场 l_i 属于 V。总出行时间定义为：

  TT_i^tot = TT_{o_i -> k_i}^g + TW_{k_i}^wait + TT_{k_i -> l_i}^a + TT_{l_i -> d_i}^g

  其中：
    TT^g     = 地面出行时间
    TW^wait  = 排队等待时间
    TT^a     = 空中出行时间

2.2 起降场布局

环境设置 8 个同时具备起降功能的垂直起降场，呈 X 型布局覆盖 40x40 km 城市
区域：

  编号   坐标 (km)      编号   坐标 (km)
  0      (5, 5)         4      (15, 25)
  1      (35, 5)        5      (25, 25)
  2      (15, 15)       6      (5, 35)
  3      (25, 15)       7      (35, 35)

乘客在地图范围内随机刷新，起点与终点由 16 个 10x10 km 网格块随机组合生成，
覆盖不同距离的出行需求。

2.3 乘客刷新与分类机制

乘客在起点被激活后，系统首先计算起点到终点的地面直线距离。对于短途出行
乘客（地面出行时间小于任何可行飞行方案），系统直接分配地面交通，乘客以
CAV 速度（20 km/h）前往终点，总出行时间仅包含地面时间，不进入起降场排队
与飞行流程。

对于中长途出行乘客，系统将为其分配起飞机场。乘客以地面 CAV 速度从起点
移动至起飞机场，进入排队队列等待空闲 eVTOL。

2.4 eVTOL 调度策略

本环境实现了区别于原论文"服务率近似"的真实调度动力学。每个时间步
（1 tick = 0.1 分钟）依次执行以下调度逻辑：

  (1) 起飞阶段：遍历所有起降场，若某起降场同时存在排队乘客与空闲 eVTOL，
      则飞机搭载一名乘客立即起飞，飞往目标起降场。飞机进入飞行状态，
      倒计时飞行剩余时间；乘客计入已完成列表，记录其总出行时间。

  (2) 调度阶段：系统扫描所有起降场的排队长度，定位当前排队人数最多的
      起降场作为目标。随后检索所有空闲 eVTOL，选择距离目标最近的 eVTOL
      前往支援。该 eVTOL 飞抵目标起降场后，若该场仍有排队乘客，则立即
      接客起飞，形成动态补位机制。

  (3) 飞行更新阶段：每个时间步更新所有飞行中 eVTOL 的剩余飞行时间，
      当剩余时间归零时，eVTOL 到达目的地起降场，重新加入空闲飞机池，
      准备执行下一任务。

上述调度逻辑在有限飞机数量下尽可能均衡各起降场的服务能力，模拟了真实
UAM 运营中的动态配车机制。

2.5 优化目标

系统目标为最小化所有乘客的平均总出行时间：

  min      (1/|N|) * sum_i TT_i^tot
  (k_i,l_i)

该问题为典型的序贯决策问题，乘客间通过共享起降场与 eVTOL 资源产生强耦合，
决策的时变性与排队动态的非平稳性使得传统静态优化方法难以适用。因此，
本文将其建模为马尔可夫决策过程（MDP），采用深度强化学习求解。


三、方法

3.1 状态空间设计（20维）

在时刻 t，智能体从环境中观测到 20 维状态向量：

  S(t) = [Q_0, ..., Q_7, F_0, ..., F_7, x_o, y_o, x_d, y_d]

  分量        含义                        维度    归一化方式
  Q_k         起降场 k 排队人数            8       Q_k / (max Q + 1)
  F_k         飞往起降场 k 的 eVTOL 数     8       F_k / (max F + 1)
  (x_o, y_o)  乘客起点坐标                 2       x / 40
  (x_d, y_d)  乘客终点坐标                 2       x / 40

F_k 的设计动机：排队人数 Q_k 反映的是当前时刻的拥堵状态，但无法预示排队
趋势。考虑两种场景：起降场 A 排队 30 人但有 30 架 eVTOL 即将到达，实际
等待时间接近于零；起降场 B 排队 20 人但无 eVTOL 即将到达，实际等待时间
很长。F_k 的引入使智能体能够预判排队演化趋势，从而做出更优的起降场选择。

3.2 动作空间裁剪

动作为七选一离散选择：

  A(t) 属于 {0, 1, ..., 6}

其中动作 a 表示选择按总出行时间（步行 + 飞行）排序后的第 a 个起降场
（0 为最近，6 为第七近）。降落机场固定为离乘客目的地最近的起降场。

3.3 奖励函数设计

奖励函数定义为已完成乘客平均总出行时间的负值：

  R(t) = -(1/|N_t|) * sum_i TT_i^tot

其中 N_t 为时刻 t 已完成出行的乘客集合，TT_i^tot 为乘客 i 的总出行时间。
该奖励设计直接对齐优化目标，不存在奖励塑形带来的目标偏差问题。

3.4 PPO 算法配置

本文采用 Proximal Policy Optimization（PPO）算法优化策略网络。策略网络为
两层 256 维全连接网络，超参数如下：

  参数              值
  学习率            5e-4（线性衰减）
  n_steps           2048
  n_epochs          10
  batch_size        64
  gamma             0.99
  GAE lambda        0.95
  clip range        0.2
  熵系数            0.03


四、实验设置

本文实验基于 40x40 km 城市区域、8 个 X 型布局起降场的仿真环境。主要参数
配置如下：

  参数                    值
  地图大小                40 x 40 km
  起降场数量              8 个
  地面 CAV 速度           20 km/h
  eVTOL 飞行速度          120 km/h
  eVTOL 载客量            1 人
  乘客规模                800 人（固定随机种子）
  eVTOL 数量              8 / 12 / 16 架
  训练步数                12 万步
  学习率                  5e-4（线性衰减）
  折扣因子 gamma          0.99
  熵系数                  0.03

对比方法：

  (1) 全部走地面：所有乘客全程乘坐 CAV，不乘坐 eVTOL。总出行时间仅包含
      地面行程。

  (2) 最短路径：每位乘客独立选择使 TT_{o_i -> k_i}^g + TT_{k_i -> l_i}^a
      + TT_{l_i -> d_i}^g 最小的起降场组合，k_i 与 l_i 可相同。该方案
      不考虑排队等待时间。

  (3) 最短路惩罚：在最短路径基础上引入排队惩罚项，将起降场选择代价修正
      为 TT^total + lambda * Q_k，其中 Q_k 为起降场 k 的当前排队人数。
      惩罚系数 lambda 取 225，该值通过对数网格搜索确定。该方案作为
      规则化的避堵基线。

  (4) 基线 PPO（12维）：采用与本文相同的 PPO 算法与网络结构，但状态空间
      仅包含 8 个起降场的排队人数与 4 维起终点坐标（共 12 维），不含
      飞往 eVTOL 数量特征。该方案用于验证本文 20 维状态设计的有效性。

评估指标：

  (1) 平均地面出行时间（AGT）
  (2) 平均等待时间（AWT）
  (3) 平均空中出行时间（AAT）
  (4) 平均总出行时间（ATT）：AGT + AWT + AAT，本文的最终优化目标
  (5) 系统完成时间：所有乘客完成出行所需的总仿真时长

六、实验结论

综合上述实验结果，可得出以下结论：

  (1) PPO 方法优于规则基线

      在三种 eVTOL 配置下，本文 PPO 方法均优于最短路径与最短路惩罚基线。
      8/12/16 架配置下相比最短路径分别降低 41.0%/40.8%/22.7%，相比最短路
      惩罚分别降低 14.8%/14.6%/13.9%。强化学习通过与环境交互学习动态
      避堵策略，优于固定规则的惩罚方法。

  (2) 排队预判特征有效

      通过消融实验对比基线 PPO（12维，不含飞往 eVTOL 数量）与本文方法
      （20维，含飞往 eVTOL 数量），AWT 在 8/12/16 架配置下分别降低
      7.3%/7.8%/10.8%。观察飞往起降场的 eVTOL 数量使模型能够预判排队
      趋势。

  (3) 运力紧张场景下模型优势更显著

      8 架 eVTOL（运力紧张）时本文方法相比基线 PPO 的 ATT 降低 6.3%，
      而 16 架（运力充足）时仅降低 4.8%。在运力紧张时排队更容易累积，
      模型通过排队预判与动态避堵获得的收益更大。

  (4) 地面-飞行分类机制有效

      本文方法在三种配置下的地面完成人数分别为 351/345/332 人，显著高于
      最短路径（均为 228 人）与最短路惩罚（301/295/280 人）。短途乘客
      被合理分配地面交通，有效缓解了起降场排队压力。


七、研究局限与未来方向

  (1) 算力受限导致训练不充分

      受限于本地 CPU 算力，训练步数仅为 12 万步，模型稳定性与最优性可能
      未达上限。未来工作将借助 GPU 资源将训练步数扩展至 50-100 万步，
      并开展多随机种子实验。

  (2) eVTOL 调度规则尚为人工设计

      调度策略基于人工设计规则，尚未实现调度与乘客选择的联合优化。未来
      工作将引入多智能体强化学习框架，实现 eVTOL 调度与乘客起降场选择的
      端到端联合优化。

  (3) 场景假设与真实运营环境存在差距

      实验基于理想化假设（固定乘客数量、固定地面速度、无空域约束等）。
      未来工作将引入动态乘客生成模型、起降场容量约束、充电约束与空域
      约束。

  (4) 算法未充分捕捉时序特征

      策略网络采用 MLP 结构，未能显式建模排队时序演化规律。未来工作将
      引入 Transformer 或 LSTM 等时序建模模块，使智能体具备对排队趋势的
      长时记忆与预测能力。

8.1 安装依赖

  pip install -r requirements.txt

  python generate_passengers.py

  python test_all_ground.py              # 全部走地面
  python shortest_path.py                # 最短路径
  python shortest_path_with_penalty.py   # 最短路惩罚

  python train.py                        # 训练基线 PPO（12维）

  python test.py                         # 测试基线 PPO（12维）
  python test_improved.py                # 测试本文方法（20维）
